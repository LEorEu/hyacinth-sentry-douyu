# Hyacinth Sentry (风信子哨兵) — 需求与设计

- 项目代号: **Hyacinth Sentry (风信子哨兵)**
- 目标用户: 斗鱼户外直播主播 + 同房间观众(只读)
- 文档状态: v0.5(2026-05-09)

---

## 1. 背景与定位

斗鱼户外主播在直播过程中面对两个具体痛点:

1. **礼物多而易漏**。高峰时段同时来好几个礼物会被后面的顶走;主播注意力可能在户外操作上,未必盯屏幕,可能没听见。
2. **弹幕多但信号低**。大量复制 +1 弹幕主播自己能注意到(量本身就是信号),**真正稀缺的是少数几条手写弹幕**——可能是问候、提问、提醒、点歌等,需要被识别并响应。

工具不是"再做一个弹幕窗口",而是替主播**筛信号 + 留底**。明确两类角色:

- **实时辅助**:看一眼就知道现在该响应什么
- **事后留档**:主播下播后翻账或导出

**形态**:Web 应用,移动端为主(主播自己手机的浏览器,放在视线随手能瞥到的地方),桌面是兼容场景。

## 2. 四 Tab 设计

整体 4 个顶层 Tab:**礼物 / 高能 / 弹幕 / 历史**。

### 2.1 礼物 Tab

**单 feed 视觉,DOM 内部分两段管理**:

- **永久段(DOM 头部)**:≥ 100 鱼翅 等效价值的礼物 + 钻粉/贵族订阅;按时间倒序,永不淡出
- **瞬时段(紧随其后)**:阈值 ≤ 价格 < 100 鱼翅的礼物;按时间倒序,**30 分钟 TTL** 后自动从 feed 移除(不动 DB)

视觉上是一个连续的时间倒序列表,两段之间有一条细虚线作为分隔提示。永久段的卡片左边框加粗 + 染红,让主播一眼可辨。

**等效鱼翅 (effective_yuchi)** 是分档与过滤的核心指标:

```
effective_yuchi = max(price_yuchi, intimacy / 10)
```

亲密度按 1:10 折算为鱼翅(用户拍板的折算系数,源自"抱元守一 1000 亲密度 ≈ 钻粉飞机 100 鱼翅"的视觉等价)。这样乾坤袋抽出的亲密度礼物能与同档真鱼翅礼物自动归到同一颜色档。

**显示规则**(主播视角):

| 礼物来源 | price_yuchi | intimacy | 显示 | 颜色档 |
|---|---|---|---|---|
| 钻粉飞机 (买) | 100 | 1000 | "100 鱼翅" | 100鱼翅档 |
| 抱元守一 (乾坤袋抽中) | 0 | 1000 | "1000 亲密度" | 100鱼翅档(同上) |
| 超大丸星 (鱼丸购) | 0 | 1000 | "1000 亲密度" | 100鱼翅档 |
| 山海有灵 (乾坤袋抽中) | 0 | 60 | "60 亲密度" | 6鱼翅档 |

**阈值过滤整列**:`effective_yuchi < 阈值` 的礼物**根本不进 feed**(永久段 / 瞬时段都不进)。阈值默认 **6 鱼翅**(办卡价值),UI 输入框可调,写入 `localStorage.prefs.gift_threshold_yuchi`。

**collector 层硬过滤**:`STORE_THRESHOLD_YUCHI = 6.0`,`effective_yuchi * count < 6` 的礼物**直接 drop,不入 DB 也不广播**。这层是为了防止 ID 自增膨胀(SQLite AUTOINCREMENT 严格单调,删老数据无效),把"赞 0.1 鱼翅"这类海量噪音从源头挡掉。catalog 完全未收录的新礼物 (`price_type=None && pid 不在 pandora 池`) 走安全网保留。

新礼物 / 订阅到来时:
- 屏幕角标闪烁 3s(默认开)
- 震动通知(待开发):≥100 鱼翅等效触发 `navigator.vibrate`,主播手机锁屏放兜里也能感觉到

订阅事件视为永久段(无价格但价值高)。

### 2.2 高能 Tab

时间倒序的卡片列表。每条显示:发送人昵称 / 内容文本 / 价格(鱼翅)/ 接收时间 / 子类型角标(如 `voiceDanmu`)。

**主播一点切换"已响应"状态**(整张卡片可点,不要小按钮):
- 点一下 → 卡片半透明 + 划线 + 右上 ✓ 角标 + 灰边框,**不消失**(主播能回看自己刚标了什么)
- 再点一下 → 撤销
- **不弹确认对话框**,误点了就再点一下
- 顶部固定显示 `未响应 N / 共 M` 计数
- **乐观更新** + 失败回滚;服务端通过 WS 广播 `done_changed`,多设备同步
- DB 列 `done_at` 存 ms 时间戳;NULL = 未响应;CSV 导出含此列

**协议层过滤**:`comm_chatmsg` 中 `cprice <= 0` 的事件 collector 直接 drop。这类事件主要是 `btype=pandora`(潘多拉宝箱抽中礼物的全房广播),主播此刻没收鱼翅(那笔钱在买宝箱时已计入 `dgb`),保留只会是重复信号。

**自动识别(留待 Phase 4,需要主播提供真实样本词表)**:点歌 / 任务两类的关键词识别 + 自动打 `auto_tag` 列。

### 2.3 弹幕 Tab

上下两栏 70/30 分栏(CSS flex,固定比例不可拖)。

#### 上方 · 实时弹幕(带昵称)

**核心机制是去重 +N,不做长度过滤、不做停用词**——基于 2026-05-09 5 分钟真实数据采样得出:该房间 80% 弹幕都是重复刷屏,但刷的是房间特色梗(主播自己的口头禅),通用停用词包几乎无收益;长度过滤反而过滤掉**有意义的短回应**("考"30 次、"爆了"7 次都是实际信号)。真正的产品价值在**合并复读 + 自然过期**。

| 维度 | 行为 |
|---|---|
| 归一化 | 小写 + 去空白 + 去标点 `、。,!?!?,.~~·…\-—` + 去尾部 `+N` |
| 去重 | 同 norm key 的弹幕合并为一条,显示 `+N` 计数徽标,每次复读把这条提到 dedup 段顶部 |
| 滑窗 | **1 分钟**没新复读则节点 expire 移除(10 秒一次 sweep) |
| 节点上限 | dedup 段最多 200 条 |
| 关键词置顶 | 主播自定义关键词(自己昵称、口头禅)命中→**单独条目**飘红 + hold 30s + 不参与去重合并 |

DOM 顺序 = `[pinned 段,最新关键词命中在前]` + `[dedup 段,最近复读在前]`,两段间细虚线分隔。

#### 下方 · 热梗(不显示昵称)

数据源 = 斗鱼自家的 `rec_barrage_hot` 帧。每帧 JSON: `{"bg": 完整原句, "hot": 热点关键词, "showTime": 10, "total": N}`。渲染成 `{N} 人在说 {hot}` + 完整原句缩略。同 `hot` 短语再来一帧→更新 N + 提到顶部;1 分钟没刷新→移除。

无 `rec_barrage_hot` 时下栏显示空——上栏的 dedup +N 已经覆盖了"现在大家在刷什么"的需求。

### 2.4 历史 Tab

按日期查询(日期选择器,默认今日):

- 当日所有**高能弹幕**(含完成状态)
- 当日所有**礼物**
- **不展示弹幕**(弹幕量大、留底意义低,且过滤后的优质流已经实时呈现)
- 支持导出 CSV(已脱敏 uid)

## 3. 通用机制

### 3.1 时间窗口(凌晨 4 点切分)

- 各 Tab 的"今日视图"以**凌晨 4 点**为切分点(贴合户外主播作息)
- DB 底层不做切分,事件表追加写入
- 各视图按 `ts` 范围查询,前端展示标题"今日(YYYY-MM-DD 04:00 起)"

### 3.2 关键词置顶(弹幕 Tab 专用)

- 主播在 UI 维护关键词列表,持久化在 localStorage(不入 DB)
- 命中时弹幕飘红 + 顶部停留默认 30s
- 与去重独立:命中的不论是否手写、是否口号,都置顶(防止"主播主播在不在"被去重掉)

### 3.3 礼物 / 大事件通知

| 通道 | 默认 | 说明 |
|---|---|---|
| 角标闪烁 | 开 | header 右上小圆点 3s 三次闪烁(已实现) |
| 震动 | 开关待加 | `navigator.vibrate`,≥100 鱼翅触发,移动端锁屏可感知 |

> **TTS 已废弃**——户外 + 主播自己手机 + 小扬声器场景下信噪比基本为零,且会被主播自己讲话盖住。改用震动 + 视觉。

### 3.4 用户角色(主播 vs 观众)

工具同时面向主播自用和同房间观众查缺补漏:

| 角色 | 区分方式 | 能力 |
|---|---|---|
| 主播(admin) | URL `?key=<DOUYU_ADMIN_KEY>` 一次写入 localStorage | 全部功能;高能 ✓ 状态切换;导出 CSV |
| 观众(read-only) | 默认 | 只读浏览四个 Tab;管理控件全部隐藏(CSS `body.admin` 类控制) |

**单机模式**:未设置 `DOUYU_ADMIN_KEY` 环境变量时,所有人均为 admin,header 不显示"主播" pill(标识无意义)。

观众"查缺补漏"采用**只读方案**:观众发现主播漏掉的高能/礼物时,直接在斗鱼弹幕里告诉主播即可,**不在工具内构建建议队列**。

### 3.5 房间状态展示

header 实时显示:
- **直播中 / 未开播** pill:来自 betard `show_status`(1=直播中 / 2=未开播),后端 8s 轮询
- **贵宾数**:来自 danmu 协议 `oni` 帧的 `vn` 字段,~5s 一推。**注意**:之前用过 betard 的"在线热度"(`hn`/`online`/`hot`),那是给观众看的虚数,无意义,已废弃
- **当日统计** pill:`鱼翅 N · 高能 M`(精简过的;之前的"礼物 N/M 鱼翅"里礼物条数无信号)

## 4. 数据模型

`events` 单表存储,SQLite WAL 模式。完整 schema 见 [db.py](./db.py)。

| 列 | 类型 | 含义 |
|---|---|---|
| `kind` | TEXT | `gift` / `superchat` / `subscription` / `chat`(不入库) / `hot_barrage`(不入库) / `vip_info`(不入库) |
| `price_yuchi` | REAL | **真鱼翅**(YUCHI 礼物的 ÷100 转换值);亲密度礼物 / 鱼丸礼物的此列为 NULL,避免污染统计总额 |
| `color` | INTEGER | 1 红 / 2 蓝 / 3 绿 / 4 橙 / 5 紫 / 6 粉,钻粉/贵族彩弹专用,**不是高能弹幕** |
| `done_at` | INTEGER | 高能"已响应"时间戳(ms);NULL = 未响应 |
| `raw` | TEXT | 原始 KV body,排查协议变更时用 |

**WS payload 额外字段**(不入 DB,只随实时推送给前端做亲密度/鱼丸特殊渲染):

| 字段 | 含义 |
|---|---|
| `intimacy` | 单位亲密度;主 catalog 的 `growthInfo.intimacy` 优先,否则取 pandora 池的 `intimacy_per_unit` |
| `price_yuwan` | 单位鱼丸价格(YUWAN 礼物专用);仅超大丸星/100鱼丸 |
| `price_type` | "YUCHI" / "YUWAN" / null(未收录) |
| `effective_yuchi` | 单位等效鱼翅 = `max(price_yuchi, intimacy/10)`;前端做颜色档与永久段判断 |

**配置项**(关键词列表、阈值、通知开关)全部放 localStorage,不入 DB。

**已知遗留**:历史 Tab(走 `/api/history` 从 DB 查)拿不到上述 4 个 ephemeral 字段,所以早期入库的乾坤袋礼物(抱元守一/千机伞 等)在历史 Tab 里只能看到名字+数量,看不到亲密度。需要时可加 `intimacy / price_type` 两列 + 一次 backfill。

## 5. 协议层关键事实

> 完整版与踩坑历史详见项目级 memory `douyu_protocol_notes.md`。这里只列最容易翻车的几条。

- **服务端:** `danmuproxy.douyu.com:8601` (TCP)。老的 `openbarrage.douyu.com` 已下线。
- **匿名(uid=0)** 即可收齐所有公开事件,无需账号(也避开了风控)。
- **多 gid 才能收齐弹幕**:实测 `[-9999, 1, 2, 3, 4, 5]` 比单 `-9999` 多约 43%。
- **价格单位三套**:
  - 礼物 catalog `price_info.price` / `pc` / `comm_chatmsg.cprice` —— 鱼翅 × 100
  - `dfobc.price`(钻粉开通)—— RMB 分,÷10 才是鱼翅
  - 订阅 `odfpbc / rdfpbc` —— 没有 price 字段(赠送 / 活动型)
- **catalog priceType**:实测只有 `YUCHI` / `YUWAN` 两种,**没有 INTIMACY**;亲密度走 `growthInfo.intimacy`。
- **乾坤袋抽中礼物**(抱元守一/千机伞 等)**不在主 gift catalog 里**,需要从 `https://www.douyu.com/japi/interact/comm/pandora/config?rid={rid}` 抓"奖品池",pid → `value` (单位亲密度) 映射。dgb 帧的 **`from=2`** 是"乾坤袋抽中"的标记(普通购买为 `from=1`)。
- **`from=2` + `gfid=0` + `pid` ≠ 0** = 乾坤袋抽出礼物的协议特征。
- **订阅事件多 gid 重复广播**:`dfobc/odfpbc/...` 实测会在多个 gid 通道同时到达,collector 用 (type, uid) + 5s 窗口 LRU 去重,避免同人开钻粉入库两条。
- **高能弹幕 = `comm_chatmsg`**,不是 `chatmsg + col≠0`(后者只是钻粉/贵族彩弹特权)。
- **`comm_chatmsg.chatmsg@=` 是嵌套 KV**,二级用 `@A=` 替代 `@=`、`@S` 替代 `/`,需要 `parse_kv` 二次解。
- **贵宾数**:`oni` 帧的 `vn` 字段(每 5 秒推一次),不是任何 betard 字段。
- 未识别 type 自动写入 `diag.log`(每 type 5 条样本封顶)+ 首次见 INFO 一条。

## 6. 已知 bug / 待办

### 6.1 历史 Tab 看不到亲密度礼物的价格信息

DB schema 没有 `intimacy / price_type` 列,亲密度信息只在实时 WS payload 里。所以历史 Tab 的乾坤袋礼物只能看到名字+数量,看不到亲密度。**修复路径**:加两列 + backfill 一次。当前优先级低,新事件实时显示是核心场景。

### 6.2 贵族开通事件路径未实证

`anbc` / `rnewbc` 在本环境从未触发,代码分支写了但**未经验证**。下次见到时,server 日志会出现 `[diag] first-seen unknown type: X`,把那行发回即可。

### 6.3 历史查询 limit 500(轻微)

`db.py:Store.query` 硬性 cap 500,繁忙日子 backfill 和历史 Tab 都受限。当前未分页,遇到再处理。

### 6.4 pandora catalog 启动时一次性拉

斗鱼活动期间换乾坤袋皮肤(改 pid)需要重启服务才能生效。每次活动周期通常较长,当前可接受。如有需要可加定时刷新(比如每小时)。
